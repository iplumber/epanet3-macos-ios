# 大型管网视窗性能说明 v1

**实施状态**：**暂不优化代码**；优化方向已记入 [`开发任务清单_v1.md`](开发任务清单_v1.md) **§6.6**（FO-1～FO-6），与 **P2-3** 对应。

## 现象

在**视窗内几何对象数量相近**的前提下，**约 9 万节点**的模型比 **1 万节点**模型在**缩放、平移**时更易卡顿。

## 原因分析（为何与「屏内可见数量」不完全一致）

当前实现里，大量工作仍与**全图规模 \(N\)**（总节点数 + 总管段数）成近似**线性**关系，而不是与**视窗内可见数量**成比例。

### 1. Metal 每帧全量绘制（主要 GPU / 提交负载）

[`MetalNetworkView.draw(in:)`](EPANET3App/EPANET3Renderer/Sources/EPANET3Renderer/MetalNetworkView.swift) 对**全部**管段、**全部**节点调用 `drawPrimitives`，顶点数随 `lineCount`、`pointCount` 线性增长（例如每管段 6 顶点、每节点 6 顶点）。  
即使用户只「看」一小块区域，GPU 仍对**整网**几何做变换与光栅化管线；视窗变小并不会自动减少提交的图元数量。

相关代码特征：

- `vertexCount: lineCount * 6`（全线）
- `vertexCount: pointCount * 6`（全节点）

### 2. 每帧 CPU 侧全量扫描（与 \(N\) 成正比）

同一 `draw` 路径中仍存在对**全部管段**的遍历，例如：

- `countLinksIntersectingVisibleRect(links:scene.links, …)`：用于判断是否走管线双 pass，**复杂度 O(links)**。
- 水泵符号：对 `scene.links` 中 `kind == 1` 的管段逐段计算屏幕长度等，**O(泵段数)**，最坏仍接近 O(links)。

### 3. 拾取与框选（交互路径上的 O(N)）

- `hitTest(viewPoint:viewSize:)`：对**所有节点、所有管段**做距离计算，**O(nodes + links)**，在鼠标移动频繁触发时放大卡顿。
- `indicesInMarquee(sceneRect:…)`：框选结束时对**全图**节点/管段筛选，**O(nodes + links)**。

### 4. 画布文字标注覆盖层（SwiftUI Canvas）

[`CanvasMapLabelsOverlay`](EPANET3App/EPANET3App/CanvasMapLabelsOverlay.swift) 在统计与收集候选时仍会遍历 `scene.nodes` / `scene.links`（虽有 `visibleNodeCount` 与上限，但**单次遍历成本仍随 \(N\) 增大**）。

### 5. 小结

| 环节 | 复杂度特征（现状） |
|------|---------------------|
| 线/点绘制提交 | 与**总**管段数、总节点数近似线性 |
| 部分每帧辅助逻辑 | O(links) 全表扫描 |
| 点击/框选 | O(nodes + links) |
| 标注候选（未做空间索引时） | O(nodes + links) |

因此：**视窗内「看起来一样密」只说明像素上重叠少，并不等于 CPU/GPU 工作量与 1 万节点同级**；9 万节点的全量提交与全表扫描仍比 1 万重一个数量级。

## 优化方案（按优先级与性价比）

### P0：渲染只做「可见子集」（收益最大）

- **思路**：在场景坐标下用**视窗矩形**（可略扩边）裁剪，只把**与视窗相交的**管段、节点写入当帧使用的 GPU buffer（或只提交对应 `vertexStart`/`vertexCount` 的连续区间 / 多段 draw call）。
- **前提**：几何在 `sceneGeometryRevision` 变化时维护**空间索引**（见下），或每帧对索引做一次**查询**得到可见索引列表（仍比绘制全量便宜）。
- **预期**：缩放平移时工作量近似 **O(可见 + 索引查询)**，大屏密网体验会接近「按视窗规模」而非「按全网规模」。

### P0：空间索引（支撑绘制、拾取、标注）

- **结构**：均匀网格、四叉树、R-tree 等，按**节点点**与**管段包围盒**建索引；几何变更时重建或增量更新。
- **用途**：
  - 可见集查询（绘制）
  - `hitTest` / 框选先查索引再细算（可先查桶再精确距离）
  - 标注层只遍历候选索引，避免整表扫描

### P1：拾取与鼠标移动降频

- `hitTest` 前用**粗网格桶**筛掉不可能命中的大部分对象。
- 鼠标移动：对 `onMouseMove` **节流**（例如 30–60 Hz）或仅在按下/拖拽时高频拾取。

### P1：`countLinksIntersectingVisibleRect` 与水泵循环

- 若已有空间索引：**只对视窗候选管段**计数与算屏幕长度，避免每帧 O(links)。

### P2：标注层

- 在已有 `clipMargin` 与数量上限基础上，用**空间索引**或**分桶**只收集视窗附近对象，避免 9 万次循环。

### P2：极端缩放下的 LOD（可选）

- 全图极缩小时：节点可聚合显示、管段可降采样或合并绘制，减少顶点与片元压力（产品需接受视觉简化）。

### P3：长期

- **GPU 端视锥裁剪**可减轻片元压力，但**顶点仍已提交**；根本仍是「少提交不可见图元」或间接绘制。
