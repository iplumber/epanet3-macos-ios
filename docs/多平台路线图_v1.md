# 多平台路线图 v1

## 1. 文档目的

记录产品层**阶段性路线**（macOS → iOS → B/S → 端云上传），并与 [`开发任务清单_v1.md`](开发任务清单_v1.md) 中的 **P0 / P1 / P2** 对齐，便于排期与里程碑复盘。

**状态**：规划记录（非已实现功能清单）。

---

## 2. 四步计划（高层）

| 阶段 | 内容 | 说明 |
|------|------|------|
| **第一步** | 完善 macOS 功能 | 以桌面端为**主战场**，在现有编辑/计算闭环上强化界面与体验。 |
| **第二步** | 完善 iOS 功能 | 在引擎与桥接稳定后，聚焦小屏布局与触控效率。 |
| **第三步** | 筹划 B/S（浏览器/服务端）版本 | 新产品线：范围、技术栈、部署与成本需单独立项；可与原生并行**预留接口形态**。 |
| **第四步** | macOS / iOS 直传文件到 B/S | 依赖服务端**上传与鉴权**能力；客户端需实现安全上传（如 `URLSession`、后台传输等）。 |

---

## 3. 与现有任务清单的对应关系

| 路线图阶段 | 对应 `开发任务清单_v1.md` |
|------------|---------------------------|
| 第一步（macOS） | **[开发任务清单](开发任务清单_v1.md) §8** 与本表；详项见 **§6.2 P1（macOS 界面强化）**；发布收尾前仍需 **§6.1 P0**。 |
| 第二步（iOS） | **§6.3 P2** 中 **P2-4（iOS 小屏布局与触控效率优化）** 等；其余 P2 项按优先级穿插。 |
| 第三步（B/S） | **本仓库当前无独立实现任务**；建议在启动前单独产出：范围说明、API 草案、部署方案。 |
| 第四步（上传） | 依赖第三步至少具备：**上传 URL、鉴权、限额与格式策略**；宜在 B/S MVP 中**首批**定义 API，避免后期返工。 |

### 3.1 推荐顺序（与清单 §6.4 不冲突）

1. **先完成 P0**（v1 基线冻结与验收），再大规模推进 P1，避免无版本锚点。
2. **P1 以 macOS 为主**（对应路线图第一步）。
3. **P2 中优先 iOS 相关项**（对应路线图第二步），再扩展结果/性能等。
4. **B/S 与上传**：在第三步、第四步启动时，单独建 **里程碑与交付物**（文档 + 仓库或子项目）。

---

## 4. 依赖与风险（简要）

**依赖**

- **第四步依赖第三步**：无接收端与策略，上传无法闭环。
- **第三步可与第一、二步并行「纸面/API 设计」**，但**落地**通常需专门投入，避免与 P1 抢同一窗口。

**风险**

- B/S 范围蔓延（只读展示 vs 在线计算 vs 多用户）→ 需尽早冻结 **MVP**。
- 上传涉及**账号、隐私、HTTPS、文件大小**；**合规与运维**需提前评估。
