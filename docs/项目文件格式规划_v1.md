# EPANET3 项目文件格式规划（`.enp` — EPANET Network Project）

> 版本：v1.0-draft  
> 日期：2026-04-15  
> 状态：需求与概要设计

---

## 1　背景与动机

### 1.1　当前状况

| 数据类别 | 现状 | 痛点 |
|---------|------|------|
| **管网模型** | EPANET 2.2/3.0 `.inp` 纯文本 | 只能存标准 EPANET 字段；读写需逐行文本解析，大型管网 I/O 慢 |
| **项目元数据** | 无持久化 | 打开文件后画布缩放/平移/显示模式/结果叠加全部丢失 |
| **显示参数** | `@AppStorage` (UserDefaults) | 全局而非逐工程；无法跟文件一起分享 |
| **扩展属性** | 不存在 | 管段材质、道路名称、分区等 GIS 属性无处记录 |
| **扩展对象** | 不存在 | 水表、消火栓、DMA 分区等水务业务对象无法表达 |
| **SCADA 映射** | `*.scada-mapping.json` 侧车文件 | 松散关联，与 INP 分离易丢失 |
| **SCADA 设备** | CSV 逐次手动导入，内存暂存 | 不随项目保存；关闭即丢 |
| **监测时序** | CSV 导入后存为 `[String: [Float]]` 字典 | 不持久化，内存开销大，无索引 |
| **计算结果** | `TimeSeriesResultStore` 纯内存 `[[Float]]` | 不持久化；大管网 × 长时序占用数百 MB RAM |

### 1.2　设计目标

1. **兼容 INP**：可无损导入/导出标准 EPANET 2.2/3.0 `.inp`；与现有工具链互操作。
2. **完整项目**：一个 `.enp` 文件包含管网模型 + 元数据 + 显示 + 扩展属性/对象 + SCADA + 计算结果，不再需要散落的侧车文件。
3. **高效存储**：二进制列式存储时序数据，比 INP 文本节省 5–10× 空间；支持内联压缩。
4. **随机查询**：按对象/时刻/参数维度直接定位，无需全量加载；典型查询 O(1) 或 O(log n)。
5. **AI 友好**：有结构化 schema、元数据表与 JSON-like 属性列，方便外部工具（Python / LLM）直接 SQL 查询。
6. **增量写入**：编辑管网、运行计算后只更新变化部分，无需整文件重写。
7. **跨平台**：macOS / iOS / Linux / Windows / Python / Web（WASM）均可读写。

## 2　格式选型：SQLite 单文件数据库

### 2.1　为什么选 SQLite

| 备选方案 | 优点 | 否决理由 |
|---------|------|---------|
| ZIP 包（多文件）| 直观 | 无法随机访问内部二进制；修改一个值需解压/重压 |
| Protocol Buffers / FlatBuffers | 极快序列化 | 非数据库；无 SQL 查询；无增量写 |
| HDF5 | 科学数据标准 | 库体积大、iOS 依赖复杂、写并发差 |
| 自定义二进制 | 理论最优 | 维护成本极高，schema 演进困难 |
| **SQLite** | **单文件、零配置、跨全平台、ACID、SQL 可查、iOS/macOS 系统内置、Python 标准库** | — |

SQLite 是 **GeoPackage**（OGC 标准）、**Apple Core Data**、**Android Room** 的底层；在水务/GIS 领域已被广泛验证。

### 2.2　文件标识

- 扩展名：`.enp`（**E**PANET **N**etwork **P**roject）
- MIME：`application/x-epanet-water-project`
- SQLite `application_id` PRAGMA：`0x454E5000`（ASCII "ENP\\0"）
- SQLite `user_version` PRAGMA：用作 schema 版本号（初始 `1`）

    init_level  REAL,
    min_level   REAL,

    diameter    REAL,
    min_volume  REAL,

CREATE INDEX idx_nodes_id ON nodes(id);

CREATE INDEX idx_links_from ON links(from_node);

#### 3.4.3　监测时序

```sql
-- 元数据
CREATE TABLE scada_ts_meta (
    series_id       INTEGER PRIMARY KEY AUTOINCREMENT,
    device_id       TEXT NOT NULL,
    kind            TEXT NOT NULL,           -- "flow" / "pressure"
    import_time     TEXT,                    -- ISO-8601
    time_start      INTEGER,                -- Unix epoch (seconds)
    time_end        INTEGER,
    sample_interval INTEGER,                -- 采样间隔（秒）
    point_count     INTEGER,
    source_file     TEXT                     -- 原始 CSV 文件名
);

-- 时序数据：列式 BLOB 存储（高密度，AI / Python 可 numpy.frombuffer 直接读）
CREATE TABLE scada_ts_data (
    series_id     INTEGER NOT NULL REFERENCES scada_ts_meta(series_id),
    chunk_idx     INTEGER NOT NULL DEFAULT 0,
    time_blob     BLOB,                     -- int32[] little-endian, 仿真秒偏移
    value_blob    BLOB,                     -- float32[] little-endian, 已换算值
    compression   TEXT DEFAULT 'none',      -- "none" / "zlib" / "lz4"
    PRIMARY KEY (series_id, chunk_idx)
);
```

**设计要点：**  
- 每条时序存为一个 `series_id`，`time_blob` / `value_blob` 为 **连续 Float32/Int32 二进制**。
- Python 读取示例：`np.frombuffer(row['value_blob'], dtype=np.float32)`。
- 分 chunk 支持超长序列分段写入（默认整条一个 chunk）。
- 可选 zlib/lz4 压缩，对平稳信号压缩比可达 3–5×。

### 3.5　Layer 4 — 计算结果

#### 3.5.1　仿真运行记录

```sql
CREATE TABLE sim_runs (
    run_id          INTEGER PRIMARY KEY AUTOINCREMENT,
    run_time        TEXT NOT NULL,           -- ISO-8601
    elapsed_wall    REAL,                    -- 墙钟耗时（秒）
    duration        INTEGER,                 -- 仿真总时长（秒）
    hydstep         INTEGER,                 -- 水力时间步（秒）
    step_count      INTEGER,                 -- 总时间步数
    node_count      INTEGER,
    link_count      INTEGER,
    flow_units      TEXT,
    headloss_formula TEXT,
    status          TEXT DEFAULT 'success',  -- "success" / "warning" / "error"
    message         TEXT,
    inp_hash        TEXT                     -- 对应 INP 快照的 SHA-256 前 16 位
);
```

#### 3.5.2　结果时序（列式 BLOB）

```sql
-- 节点结果
CREATE TABLE sim_node_ts (
    run_id      INTEGER NOT NULL REFERENCES sim_runs(run_id),
    param       TEXT NOT NULL,               -- "pressure" / "head" / "demand" / "tank_level"
    chunk_idx   INTEGER NOT NULL DEFAULT 0,
    -- 时间轴（整个 run 共享，此处冗余存一份或引用 run 级别）
    time_blob   BLOB,                        -- int32[] 时间点（秒）
    -- 数据：row-major float32[step_count × node_count]
    data_blob   BLOB,
    compression TEXT DEFAULT 'none',
    PRIMARY KEY (run_id, param, chunk_idx)
);

-- 管段结果
CREATE TABLE sim_link_ts (
    run_id      INTEGER NOT NULL REFERENCES sim_runs(run_id),
    param       TEXT NOT NULL,               -- "flow" / "velocity" / "headloss" / "status"
    chunk_idx   INTEGER NOT NULL DEFAULT 0,
    time_blob   BLOB,
    data_blob   BLOB,
    compression TEXT DEFAULT 'none',
    PRIMARY KEY (run_id, param, chunk_idx)
);
```

**存储效率对比（10,000 节点 × 240 步 × 4 参数）：**

| 方案 | 存储 | 加载 |
|------|------|------|
| 当前内存 `[[Float]]` | ~37 MB RAM（不持久化）| 需重新计算 |
| INP 文本输出 | ~200 MB 磁盘 | 逐行解析，秒级 |
| **BLOB + zlib** | **~8–15 MB 磁盘** | **mmap/numpy 直读，ms 级** |

**随机访问：** 查询某个节点某个参数的时序，只需定位对应 `(run_id, param)` 行的 BLOB，再按 `node_index × sizeof(float)` 偏移切片。无需加载全部结果。

### 3.6　Layer 5 — 附件与原文

```sql
-- 保留原始 INP 全文（方便回退、差异比对、与外部工具交换）
CREATE TABLE inp_source (
    version     INTEGER PRIMARY KEY,         -- 递增；每次保存 bump
    saved_at    TEXT,
    content     TEXT,                        -- INP 全文（UTF-8）
    content_hash TEXT                        -- SHA-256 前 16 位
);

-- 项目附件（设备 CSV 原文、照片、报告 PDF 等）
CREATE TABLE attachments (
    name        TEXT PRIMARY KEY,
    mime_type   TEXT,
    data        BLOB,
    added_at    TEXT
);
```

## 4　INP 兼容策略

### 4.1　导入（`.inp` → `.enp`）

```
  .inp 文件
      │
      ▼
  InpFileTextReader / EPANET C Engine
      │
      ├─→ Layer 1 (model tables)：逐节解析写入 nodes/links/patterns/...
      ├─→ Layer 5 (inp_source)：保存原文快照
      └─→ Layer 0 (meta)：从 [OPTIONS] 提取 flow_units 等
```

1. 用现有 `EpanetProject.load()` 加载引擎模型
2. 遍历引擎对象，写入 SQLite `nodes` / `links` 等表
3. 将 INP 原文存入 `inp_source`

### 4.2　导出（`.enp` → `.inp`）

1. 从 SQLite 读出 Layer 1 表，重建 EPANET C 工程
2. 调用 `EpanetProject.save()` 输出标准 `.inp`
3. 扩展属性（Layer 2）和 SCADA（Layer 3）不写入 INP（标准不支持）

### 4.3　混合模式（日常工作流）

```
 用户打开 .enp
     │
     ├─ 引擎仍从 Layer 1 或 inp_source 构建 C 工程
     ├─ 编辑属性 → 双写引擎 + SQLite
     ├─ 运行计算 → 结果直接写 Layer 4 BLOB
     └─ 保存 → 仅更新变化的 SQLite 行（增量，< 10ms）
```

## 5　AI / 分析友好设计

### 5.1　SQL 可直接查询

```sql
-- Python 中直接分析
import sqlite3, numpy as np

conn = sqlite3.connect("project.enp")

-- 查询所有压力设备及其映射的 EPANET 节点
SELECT d.device_id, d.name, m.epanet_id
FROM scada_devices d
LEFT JOIN scada_mapping m ON d.model = m.device_model AND d.kind = m.kind
WHERE d.kind = 'pressure';

-- 读取最近一次计算的压力时序
row = conn.execute("""
    SELECT data_blob, compression
    FROM sim_node_ts
    WHERE run_id = (SELECT MAX(run_id) FROM sim_runs)
      AND param = 'pressure'
""").fetchone()
data = np.frombuffer(row[0], dtype=np.float32).reshape(step_count, node_count)
```

### 5.2　结构化元数据

- `project_meta` 键值对提供给 LLM 上下文窗口：管网规模、单位制、坐标系。
- `custom_fields` 注册表让 AI 知道有哪些扩展字段可查。
- `sim_runs` 记录每次计算的参数与结果状态，AI 可比较不同方案。

### 5.3　标准嵌入友好

`.enp` 本质是标准 SQLite，可被：
- **Python** `sqlite3` / `pandas` / `duckdb` 直接读取
- **LangChain** SQL Agent 自然语言查询
- **R** `RSQLite` 直接读取
- **Node.js** `better-sqlite3` 读取
- **任何支持 SQLite 的 BI 工具**（Metabase、Grafana）

## 6　性能设计

### 6.1　BLOB 列式存储

时序数据以参数为粒度存储（而非以时间步为行），这意味着：

- **查询单个节点全时段压力**：读一个 BLOB → 按 `node_index` 偏移切片 → O(1)
- **查询某时刻全网压力**：读同一个 BLOB → 按 `step * node_count` 偏移 → O(1)
- **对比多次计算**：并行读取不同 `run_id` 的 BLOB → 无锁

### 6.2　压缩

| 方案 | 压缩比 | 速度 | 建议场景 |
|------|--------|------|---------|
| none | 1× | 最快 | 小管网 / 调试 |
| zlib (level 6) | 3–5× | 中等 | 默认存储 |
| lz4 | 2–3× | 极快 | 实时追加 / SCADA 时序 |

### 6.3　增量保存

SQLite 的 WAL 模式天然支持增量写入：

```swift
// 编辑某节点高程
try db.execute("UPDATE nodes SET elevation = ? WHERE id = ?", [newElev, nodeId])
// 仅写一个 page（4KB），无需整文件重写
```

对比当前 INP 保存需要**整文件序列化**，`.enp` 的保存耗时从 **数百毫秒降到 < 10ms**。

### 6.4　内存映射

- 计算结果的 BLOB 可通过 SQLite 的 `sqlite3_blob_open` **零拷贝 mmap 读取**
- 浏览时间轴时，只需 seek 到对应时间步的偏移读取一行浮点数组
- 大管网下内存占用从 **数百 MB 降到 < 10 MB**（按需加载）

## 8　文件格式版本演进策略

```
user_version = 1   初始版本（Layer 0–5 全部表）
user_version = 2   新增水质扩展表（示例）
user_version = N   ...
```

**迁移规则：**
1. 打开时读取 `user_version`
2. 若 < 当前版本，依次执行迁移脚本（`ALTER TABLE` / `CREATE TABLE`）
3. 若 > 当前版本，弹窗提示"此文件由更新版本创建，部分功能可能不可用"，只读打开
4. 迁移前自动备份为 `.enp.bak`

## 9　与现有代码的集成点

| 现有模块 | 集成方式 |
|---------|---------|
| `EpanetProject` (C Bridge) | 引擎仍从 `.inp` 文本或内存状态工作；`.enp` 在 Swift 层与引擎之间做桥接 |
| `AppState.timeSeriesResults` | 可从 BLOB 懒加载，也可在计算时直接写 BLOB 而不全量驻留内存 |
| `AppState.scadaMapping` | 从 `scada_mapping` 表加载，替代 `.scada-mapping.json` |
| `AppState.scadaDeviceCatalog` | 从 `scada_devices` 表加载，替代每次手动导入 CSV |
| `AppState.scadaObserved*ByDeviceId` | 从 `scada_ts_data` BLOB 加载 |
| `MetalNetworkCoordinator` (scale/pan) | 保存到 `display_settings` |
| `InpPreservingSaver` | `.enp` 模式下不再需要文本补丁；INP 导出时走 `EpanetProject.save()` |

## 10　安全与完整性

- **ACID 事务**：编辑和保存在 SQLite 事务中，崩溃不损坏文件
- **WAL 模式**：计算线程写入结果时，UI 线程可同时读取已有数据
- **备份**：`.enp` 是单文件，复制即备份；版本迁移前自动 `.bak`
- **校验**：`inp_source.content_hash` 可检测 INP 与模型表是否同步
- **权限**：无网络依赖，无外部服务调用，离线可用

## 附录 B：BLOB 二进制编码约定

| 字段 | 编码 |
|------|------|
| `time_blob` | `Int32[]` little-endian，每元素为仿真秒 |
| `value_blob` | `Float32[]` little-endian |
| `data_blob`（结果） | `Float32[step_count × object_count]` row-major little-endian |
| `compression = "zlib"` | 整个 BLOB 经 zlib deflate；读取时先 inflate |
| `compression = "lz4"` | LZ4 frame 格式 |

## 附录 C：Python 快速访问示例

```python
import sqlite3
import numpy as np

conn = sqlite3.connect("my_network.enp")

# 1. 管网基本信息
meta = dict(conn.execute("SELECT key, value FROM project_meta").fetchall())
print(f"项目: {meta['title']}, 单位: {meta['flow_units']}")

# 2. 节点列表
nodes = conn.execute("SELECT id, x, y, elevation FROM nodes").fetchall()

# 3. 最新一次计算的压力时序
run = conn.execute("SELECT run_id, step_count, node_count FROM sim_runs ORDER BY run_id DESC LIMIT 1").fetchone()
run_id, steps, nc = run

row = conn.execute(
    "SELECT data_blob FROM sim_node_ts WHERE run_id=? AND param='pressure'",
    (run_id,)
).fetchone()
pressure = np.frombuffer(row[0], dtype=np.float32).reshape(steps, nc)

# 4. SCADA 实测时序
ts = conn.execute("""
    SELECT m.device_id, d.value_blob
    FROM scada_ts_data d JOIN scada_ts_meta m ON d.series_id = m.series_id
    WHERE m.kind = 'pressure'
""").fetchall()
for dev_id, blob in ts:
    values = np.frombuffer(blob, dtype=np.float32)
    print(f"设备 {dev_id}: {len(values)} 点, 均值={values.mean():.2f}")
```
